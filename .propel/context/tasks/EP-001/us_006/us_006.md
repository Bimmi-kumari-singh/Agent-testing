## Story Title
   * Account Activation via Verification Link

## Description
  * As a new user who has registered an account, I want to activate my account by clicking a verification link so that my account status transitions from Pending Verification to Active and I can access authenticated features.

## Acceptance Criteria
1. Given a user record exists with status "Pending Verification" and a valid, unused verification token generated within the configured TTL (24 hours), When the user requests the verification endpoint with that token (GET /verify?token={token}), Then:
   - The system validates the token and the associated user.
   - The token is marked as used/invalidated (cannot be reused).
   - The user account status is atomically transitioned from "Pending Verification" to "Active" in the database.
   - An audit log entry is created with timestamp, user_id, token_id, action="account_activation", and initiating IP.
   - The endpoint responds with HTTP 302 redirect to the front-end confirmation UI (/verify/confirmed?status=success) for browser flows, or returns HTTP 200 with JSON { "status":"success", "message":"Account activated" } for API clients.

2. Given a user clicks a verification link with a token that has expired (older than 24 hours TTL), When the verification endpoint processes the token, Then:
   - The system rejects activation and marks/flags the token as expired.
   - The user account remains in "Pending Verification".
   - The endpoint responds with HTTP 302 redirect to the front-end error UI (/verify/error?status=expired) with user-friendly messaging and an action to resend verification email; API clients receive HTTP 410 Gone with JSON { "status":"expired", "message":"Verification token expired" }.
   - A log entry is recorded for expiry event.

3. Given a user clicks a verification link with an invalid or malformed token, When the verification endpoint processes the token, Then:
   - The system does not change any account state.
   - The endpoint responds with HTTP 302 redirect to the front-end error UI (/verify/error?status=invalid) with messaging to request a new verification email; API clients receive HTTP 400 Bad Request with JSON { "status":"invalid", "message":"Invalid verification token" }.
   - An audit/log entry records the invalid attempt (include IP and request headers).

4. Given a user clicks a verification link whose token has already been used (idempotent case), When the verification endpoint processes the token, Then:
   - The system recognizes the token as used.
   - If the account is already Active, no state change occurs.
   - The endpoint responds with HTTP 302 redirect to confirmation UI (/verify/confirmed?status=already_confirmed) or HTTP 200 { "status":"already_confirmed" } for API clients.
   - No duplicate activations occur; an audit entry notes the idempotent verification attempt.

5. Given two near-simultaneous verification requests for the same valid token, When both requests are processed, Then:
   - The system enforces atomicity via a database transaction/row-level lock or equivalent mechanism: only the first request will perform the activation and token invalidation.
   - The second request receives the idempotent outcome (redirect to confirmation / status=already_confirmed).
   - System remains consistent (no duplicate audit entries for activation; token used flag set exactly once).
   - Response times remain within acceptable threshold (per NFR) for interactive flows.

## Edge Cases
- What happens when the token expires exactly at time of verification?
  - Handling: The verification logic checks token expiry at validation time. If TTL elapsed, treat as expired — do not activate; redirect to /verify/error?status=expired; log with precise timestamp and provide resend link. If client is API, return 410 Gone.

- How does system handle account already Active when a verification link is used?
  - Handling: Respond idempotently with confirmation (status=already_confirmed). Do not change account status. Log the attempted re-activation. Offer the user a link to sign in.

- How does the system handle transient failures (DB or token store down) during verification?
  - Handling: Return HTTP 503 Service Unavailable for API clients or redirect to /verify/error?status=service_unavailable for browser flows; include retry/resend guidance. Ensure no partial activation occurs; use transactions to avoid inconsistent states. Raise an operational alert for investigation.

Notes:
- Token TTL is assumed to be 24 hours for acceptance criteria; implementation should read TTL from configuration (config-driven) and tests must validate the configured TTL behavior.
- Backend endpoint must support both browser redirect flow (HTTP 302 to front-end confirmation/error screens) and API flow (HTTP 200/4xx/410 with JSON) so the verification link can be implemented as either direct server-side redirect link or via a short-lived front-end route that calls the API.
- Security considerations: token must be single-use, stored hashed or opaque in token store, and misuse/abuse attempts must be rate-limited. Ensure logs do not contain raw token values (store only token id/hash in logs).

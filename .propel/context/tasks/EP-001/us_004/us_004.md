## Story Title
   * Generate Email Verification Token

## Description
  * As a newly-registered user, I want the system to generate and persist a single-use email verification token with a configurable TTL and configurable rotation behavior, so that I can verify my email address securely and the system enforces expiry and single-use semantics.

## Acceptance Criteria
  1. Given a newly-created user in state "PENDING_VERIFICATION", When the system generates an email verification token (automatic on registration or via explicit resend), Then:
     - a token record is persisted with token_hash (salted), created_at, expires_at = created_at + configured_TTL, used_at = null, revoked = false;
     - the plaintext token is NOT stored in the DB but is sent to the configured email service in a verification link;
     - response to the originator is HTTP 202 Accepted (or 201 Created for synchronous creation endpoint) and contains no plaintext token;
     - an audit event is created containing user_id, token_record_id (internal), generation_time, requester (system or user action).
  2. Given rotation_enabled = true and there is an existing active token, When a user requests 'resend verification token', Then:
     - the existing active token(s) for that user are atomically marked revoked=true with revoked_at timestamp;
     - a new token is created and persisted with fresh expires_at computed from now() + TTL;
     - the new token is sent to the email delivery service and only the new token will be accepted for verification.
  3. Given rotation_enabled = false and there is an existing active token, When a user requests 'resend verification token', Then:
     - no new token is generated; the existing active token is re-sent to the email delivery service;
     - expires_at is not extended by the resend operation;
     - the system returns HTTP 200 OK indicating resend queued/sent.
  4. Given a verification attempt with a plaintext token, When the system receives the token, Then:
     - the system validates by comparing hash(token_plaintext + salt) with stored token_hash using constant-time comparison;
     - if token exists AND revoked=false AND used_at is null AND now() <= expires_at, mark token.used_at = now(), set user.status = ACTIVE, emit audit log, return HTTP 200 OK;
     - if token is expired, return HTTP 410 Gone and error code ERR_VERIFICATION_TOKEN_EXPIRED;
     - if token already used or revoked, return HTTP 410 Gone and error code ERR_VERIFICATION_TOKEN_USED.
  5. Given the generator detects a token collision with an existing active token hash, When a collision is detected, Then:
     - retry generation up to MAX_GENERATION_RETRIES (configurable, default 5);
     - if still colliding after retries, fail token creation, return HTTP 500 with structured error, emit metric/alert for ops.

## Edge Cases
   * What happens when TTL is set to 0 or invalid?
     - System must validate configuration on startup and reject non-positive TTL values. If TTL is 0 due to misconfiguration, token generation must fail with a clear operational error and raise an alert. In test environments, allow TTL override for deterministic tests.
   * How does system handle concurrent resend requests (race conditions)?
     - Use DB transactions or row-level locking when creating/revoking tokens. When rotation_enabled=true operations must revoke and create in a single transaction to avoid multiple valid tokens. On transient DB lock failures, retry a limited number of times; on persistent conflicts return HTTP 409 Conflict with retry guidance.
   * How does the system behave when email delivery fails after token persistence?
     - Token remains persisted. The token creation operation returns success (202) indicating token created and email queued. Background delivery retries should occur with exponential backoff. If permanent delivery failure is returned, record a delivery failure metric and notify ops; provide UI/UX messaging to the user to update email or try again.

## Story ID
   * ID Format: us_001

## Story Title
   * Create User Account (Backend API)

## Description
  * As an unauthenticated visitor, I want to register a new user account by supplying an email and password, so that an account record is created in a Pending Verification state and a verification token/email is generated to confirm ownership of the email address.

## Acceptance Criteria
  1. Given a valid registration request (email and password compliant with policy), When the client POSTs to /api/v1/users/register, Then the API responds 201 Created with JSON containing:
     - userId (uuid)
     - email (masked, e.g., j***@domain.com)
     - accountStatus: "PENDING_VERIFICATION"
     - createdAt (ISO-8601)
     - message: "Verification email queued"
     And the database contains:
     - a user row with status = "PENDING_VERIFICATION"
     - password hashed with the configured algorithm (salted + hashed)
     - a verification token record linked to the user with expiry exactly 24 hours after issuance.
  2. Given an email that already exists in any state, When the client attempts to register with that email, Then the API responds 409 Conflict and returns body { code: "EMAIL_ALREADY_EXISTS", message: "An account with this email already exists." } and does not create any additional user or token.
  3. Given invalid input (email not RFC-5322 compliant or password fails policy: min length, required complexity, or exceeds max length), When the client posts data, Then the API responds 400 Bad Request with field-level validation errors (e.g., { email: "Invalid email format" } or { password: "Password does not meet complexity requirements" }) and no user or token is created.
  4. Given that the user and verification token are persisted but the email delivery provider is unavailable, When the system attempts to send the verification email, Then the API still returns 201 Created, the token is persisted, an email-send job is enqueued for retry, an ERROR log with correlation id is produced, and a metric/event "verification_email_send_failure" is emitted. The API consumer is not required to wait for email delivery to receive 201.
  5. Given normal operating conditions, When a registration request is processed, Then the server-side processing time target is <= 500ms at the 95th percentile and automated tests confirm password is stored only as salted+hashed (no plaintext), and that verification tokens expire 24 hours after generation.

## Edge Cases
   * What happens when the provided email is syntactically valid but points to a domain without MX records?
     - The system accepts the registration and creates the account in PENDING_VERIFICATION. The verification email is attempted; if delivery fails, it will be retried according to the email retry policy. The user can request a resend via the resend flow. This avoids blocking registrations for temporary DNS issues.
   * How does the system handle extremely long passwords or hashing library limits?
     - The system enforces a documented maximum password length (e.g., 128 characters). If exceeded, return 400 with "PASSWORD_TOO_LONG". Do not attempt to hash passwords longer than the secure-hash library supports to prevent resource exhaustion or errors.
   * What happens when the DB transaction fails after the API has validated input but before token persistence?
     - The system rolls back the transaction, returns 500 Internal Server Error with a correlation id, logs full server error details internally, and does not enqueue email or create any partial user records. Clients may retry; uniqueness checks prevent duplicates from concurrent retries.

Notes / Implementation Considerations:
- Endpoint: POST /api/v1/users/register
- Request payload example: { "email": "user@example.com", "password": "P@ssw0rd!" }
- Security: Do not log raw passwords. Use secure hashing (Argon2 or bcrypt with recommended parameters). Store salts and hashed output only.
- Data model: users table (id, email, password_hash, status, created_at, updated_at), verification_tokens table (token, user_id, expires_at, created_at, consumed_at).
- Observability: Emit events/metrics for registration_success, registration_validation_failure, registration_duplicate_email, verification_email_send_failure.
- Tests: Unit tests for validation, integration tests for DB persistence and token creation, contract test for expected JSON response, and an acceptance test for email enqueue behavior when provider returns error.
- Performance target: 95th percentile API processing time <= 500ms under baseline load; include in non-functional verification.

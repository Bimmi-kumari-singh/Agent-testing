## Story Title
   * Send Verification Email via Provider

## Description
  * As the User Registration backend service, I want to send verification emails via the configured email provider with SLA guarantees and retry logic for transient failures, so that users reliably receive verification links and accounts can be activated securely and timely.

## Acceptance Criteria
  1. Given a valid verification token and target email stored in the system, When the registration service triggers send (either as part of initial registration or explicit resend), Then the service enqueues an outbound email delivery job and responds to the caller within 500ms with HTTP 202 Accepted (or equivalent async acknowledgement) and a job ID.
  2. Given the provider returns an immediate success (HTTP 2xx) on the initial call, When the provider acknowledgement is received, Then the system updates the verification_email record: status = SENT, provider_message_id set, sent_at timestamp persisted, and the metric email_verification_send_latency recorded as time between enqueue and provider success; end-to-end should complete within 120 seconds for the normal path.
  3. Given a transient provider failure (network timeout, HTTP 5xx, or 429 rate-limit treated as transient), When a delivery attempt fails, Then the system retries automatically up to 3 additional attempts using exponential backoff intervals (30s, 2m, 10m), logs each attempt and provider response, emits attempt_count metric, and on success marks status = SENT; if all attempts fail, marks status = FAILED_RETRY_EXHAUSTED, records last error, and emits an operational alert/metric.
  4. Given a permanent provider rejection (HTTP 4xx indicating invalid recipient, suppressed address, or blocked content), When provider returns permanent error, Then the system does NOT retry, marks verification_email status = FAILED_PERMANENT, stores provider error code/message, increments failure metrics, and surfaces a resolvable error in the resend UI/API explaining the permanent failure.
  5. Given multiple send/resend requests for the same verification token within a 10 minute dedup window, When duplicates are received, Then the system deduplicates using token + idempotency key and avoids sending duplicate provider messages (limit to 1 outbound send per dedup window unless explicit user-initiated resend after failure), and returns the existing job ID/status to the caller.

## Edge Cases
   * What happens when the verification token is expired?
     - Handling: The system must not send a verification email. The send request returns HTTP 400 with a clear error "verification token expired" to the calling service. The user-facing flow must trigger a new token generation flow (dependency: us_002). The failed attempt is logged for auditing.
   * How does system handle provider permanent rejection (e.g., 4xx invalid address) vs transient errors?
     - Handling: Permanent 4xx errors are recorded as FAILED_PERMANENT and not retried. Transient errors (5xx, connection timeouts, 429 treated as transient per provider guidance) are retried per policy (30s, 2m, 10m). On retry exhaustion mark FAILED_RETRY_EXHAUSTED and alert Ops.
   * What happens when the email provider is fully down or queue backlog grows (backpressure)?
     - Handling: If outbound queue latency exceeds SLA threshold (e.g., average queue-to-send latency > 120s) or queue size surpasses configured high-water mark, the system should: (1) stop accepting non-critical mass resend requests and return 503 with Retry-After for API callers, (2) emit alert to operations, and (3) persist retryable jobs to durable queue (e.g., persistent job store) for deferred processing once provider recovers.

## Story Title
   * Reproducible Developer Environment

## Description
  * As a developer onboarding to the project, I want a Docker-based reproducible developer environment, example environment files, and local run scripts, so that I can set up a consistent local development environment quickly and reliably, reduce environment-related bugs, and shorten onboarding time.

## Acceptance Criteria
  1. Given a developer has cloned the repository and has Docker and Docker Compose (or Docker Desktop) installed, When they run the provided top-level local start script (./dev/start.sh or docker-compose -f docker/dev.yml up --build), Then the required containers (application service, database, cache, and any sidecars) start successfully within 3 minutes, each service's health endpoint returns OK, database migrations are applied automatically, seed data is loaded, and the main application is reachable at http://localhost:3000 (or documented port) with a successful smoke-test response.
  2. Given the repository contains .env.example and any service-specific env samples, When the developer copies .env.example to .env and runs ./dev/validate_env.sh, Then the script validates required environment variables, prints a clear list of any missing or malformed variables, and exits with non-zero status if validation fails; .env.example contains placeholders only and no production secrets.
  3. Given Docker is not available or the Docker daemon is not running on the developer machine, When the developer executes the local start script, Then the script detects the missing/disabled Docker, prints OS-specific installation or troubleshooting steps (including links and commands for Linux, macOS, and Windows/WSL), and exits with a non-zero status code without attempting destructive actions.
  4. Given a developer re-runs the setup or start scripts, When they re-run ./dev/start.sh or ./dev/recreate.sh, Then the scripts behave idempotently: existing containers are reused or recreated cleanly on request, database migrations are idempotent (no duplicate schema objects or duplicate seed records), and the output logs clearly show actions taken and any manual steps required.
  5. Given the provided Dockerfile(s) and docker-compose files, When a developer builds the images locally using the documented build command, Then the Dockerfiles pin base image tags or digests, the build completes without requiring undocumented secrets or external interactive input, and the README documents the exact build args and the expected local image tags so CI can reproduce the build.

## Edge Cases
   * What happens when a required port is already in use?
     - The start script checks port availability before launching containers. If a conflict exists, the script prints the conflicting port, the PID and command using it (where available), suggests an alternate free port, and provides a one-line command to stop the conflicting process if the developer agrees. If the conflict cannot be resolved automatically, the script exits with a clear message and next steps.
   * How does the system handle missing Docker or insufficient permissions?
     - The scripts detect missing Docker client or daemon early. They print actionable, OS-specific instructions (install Docker Desktop, start docker.service, enable WSL2 integration), suggest commands to add the user to the docker group where applicable, and exit with a non-zero status. The README includes troubleshooting steps and links to vendor docs.
   * What happens if .env.example contains accidentally committed secrets or extremely long tokens (secret leakage)?
     - A validation check flags potential secrets in .env.example and prevents merging via CI pre-commit check. The story includes a remediation script and instructions to rotate and remove secrets. The .env.example must only contain non-sensitive placeholders; a developer-level .env.local or a secrets manager is recommended for real credentials.

Notes / Implementation Guidance (non-normative, for devs & reviewers)
- Provide top-level scripts under /dev: start.sh, stop.sh, recreate.sh, validate_env.sh, and README.dev.md with step-by-step onboarding instructions and troubleshooting.
- Provide docker/development docker-compose overrides at docker/dev.yml and Dockerfile.dev where appropriate; ensure local builds favour development-specific caching and fast feedback.
- Seed data loader and migration runner should be idempotent and callable from scripts (e.g., ./dev/run_migrations.sh).
- Add CI checks: validate .env.example has no secrets, validate Dockerfile pins base image tags/digests, and smoke-test that docker/dev.yml can be brought up in CI-in-a-container builder environment.
- Add basic health endpoints for services to enable deterministic smoke tests (e.g., /healthz).
- Estimate: sized to be <= 5 story points (one developer, ~1-3 days), split into smaller tasks if unknowns are discovered (e.g., Windows/WSL-specific issues—create a spike).
- Acceptance of this story enables other EP-TECH stories to rely on reproducible local environment artifacts.

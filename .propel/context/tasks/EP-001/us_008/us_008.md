## Story Title
   * Handle Duplicate Registrations and Race Conditions

## Description
  * As a system/operator responsible for user registration, I want registration endpoints and persistence to be idempotent and protected by database-level uniqueness and proper error handling, so that simultaneous or repeated registration attempts for the same email do not create duplicate accounts, do not produce duplicate verification emails, and return deterministic, safe responses to clients.

## Acceptance Criteria
  1. Given no existing account for email "alice@example.com", When a client submits POST /api/v1/register with a valid payload (email normalized, password meets policy), Then:
     - The API responds 201 Created with JSON containing user_id, account_status = "PendingVerification", and location header to user resource.
     - A single user row is persisted with email stored canonicalized (lowercase, trimmed).
     - A single verification token is persisted with expiry (configurable default, e.g., 24 hours).
     - The password is stored hashed using the configured algorithm (testable: stored value != plaintext and verification function validates).
     - Exactly one verification email is enqueued/sent and an audit log entry is emitted.

  2. Given an existing account for email "alice@example.com" (status PendingVerification or Active), When the client re-submits the same registration payload for that same email, Then:
     - The API responds 200 OK (idempotent success), with JSON containing the existing user_id and account_status.
     - No new user row is created; database count for that canonical email remains unchanged.
     - No new verification token is created and no verification email is sent automatically.
     - The response includes a field explaining the account already exists and next recommended action (e.g., "resend verification" or "login/reset password").

  3. Given two or more concurrent valid registration requests for the same canonical email, When the server processes these requests in parallel, Then:
     - Exactly one transaction successfully creates the user row and verification token and results in a 201 Created to that caller.
     - Other concurrent requests must not create duplicate rows; they should detect the existing record (e.g., catch DB unique constraint violation) and return 200 OK with the existing user_id and account_status.
     - Only one verification email is sent/enqueued. The system uses an idempotent send mechanism (e.g., verification_email_sent flag, dedupe key, or single-send background job) to prevent duplicates.
     - Post-condition: database contains exactly one row for the canonical email; logs include handling of the unique constraint or dedupe event.

  4. Given a DB unique constraint violation is raised during registration, When the application catches the constraint exception, Then:
     - The application performs a fresh lookup by canonical email within a new/clean transaction.
     - The API returns 200 OK with the existing user's id and status.
     - No partial/duplicate records remain; any half-completed transaction is rolled back or compensated.
     - An audit log entry is emitted describing the uniqueness conflict and resolution path.

  5. Given the registration transaction completes but the email delivery provider is unavailable, When the send attempt fails, Then:
     - The API still returns 201 Created (user persisted and token stored).
     - The email send failure is recorded (error code, timestamp) and a retry is scheduled according to retry policy (e.g., exponential backoff, max attempts).
     - Monitoring/alerting metrics are emitted for failed sends and retry attempts.
     - The system does not roll back user creation because persistence is authoritative.

## Edge Cases
   * What happens when email differs only by case or surrounding whitespace?
     - System behavior: Normalize email (trim whitespace, lower-case domain+local part if policy requires) before uniqueness check and persistence. Any registration that differs only by case/whitespace maps to the same canonical email and will not create a duplicate account. Acceptance tests must verify trim+lowercase normalization.

   * How does the system handle different payload data (e.g., different displayName) submitted for the same email?
     - System behavior: Email is the uniqueness key; registration with the same canonical email will not create a new account even if displayName differs. The API returns the existing account details. If the user intends to change profile data, they must verify and use the update profile flow. Any attempted automatic merging of different metadata is out of scope for this story.

   * What happens on a partial failure where DB commit succeeds but post-commit actions (email send) fail repeatedly?
     - System behavior: Persist user and token atomically; email sending must be retried via a durable job/queue. The system records email send failures and exposes an admin/operator view or alert. The system must not rollback the committed user record due to external email failures.

Notes and Implementation Guidance (non-normative):
- Implement a DB-level unique index on canonical_email (create migration in US_DB_001). Enforce canonicalization consistently in the API and any background jobs.
- Registration handler should follow this pattern:
  1. Normalize input (email canonicalization, password policy check).
  2. Begin DB transaction and attempt insert.
  3. If insert succeeds: commit, persist token, enqueue email send, return 201.
  4. If insert fails due to unique constraint: do a SELECT by canonical_email, return 200 with existing user info (do not create token or new email send).
  5. Ensure that email sending is idempotent (flag on user: verification_email_sent timestamp or a dedupe key in the email job).
- Add automated tests to simulate concurrent requests (integration test that fires multiple parallel POSTs and asserts single row and exactly one email send).
- Emit metrics and audit logs for duplicate attempts, unique constraint handling, and email send failures for observability and postmortem.

## Story ID

   * Story ID: us_007

## Story Title
   * Resend Verification Flow with Rate Limiting

## Description
  * As an unverified user (or someone requesting a verification resend for their email), I want to request a resend of the verification email with clear messaging and abuse protections (rate limits, anti-enumeration), so that I can receive a new verification link when needed while the system prevents spam, protects other users, and provides actionable feedback.

## Acceptance Criteria
  1. Given an account in Pending Verification and a valid resend request (POST /api/v1/verification/resend with email),
     When the request is received,
     Then the system generates a new verification token (or rotates/invalidate previous), persists it with a 24-hour expiry, enqueues an email to the configured email provider, returns HTTP 200 with JSON { "message": "Verification email sent. Check your inbox." }, and records an audit log entry including timestamp, requester IP, user-agent, and email hash.

  2. Given a requester has made 3 resend attempts within the past 60 minutes or 10 attempts within the past 24 hours,
     When an additional resend request is made,
     Then the system returns HTTP 429 with JSON { "error":"rate_limit_exceeded", "message":"Too many resend requests. Try again in X minutes.", "retry_after_seconds": <seconds> }, does not enqueue an email, increments the rate-limit metric, and emits an audit event.

  3. Given an unauthenticated resend request for an email that does not exist,
     When the request is processed,
     Then the API returns HTTP 200 with a generic message "If an account exists, a verification email has been sent." (same message used for existing accounts for unauthenticated callers), does not create tokens, and logs the event for monitoring without exposing existence to the client.

  4. Given the account is already Verified/Active and the request is authenticated by the account-holder,
     When a resend request is received,
     Then the API returns HTTP 400 with JSON { "error":"account_already_verified", "message":"This account is already verified." }, and no email or token is generated. For unauthenticated callers requesting by email, return the generic HTTP 200 message to avoid enumeration.

  5. Given an email provider transient failure on send,
     When a resend is requested,
     Then the API returns HTTP 202 Accepted with JSON { "message":"Request accepted; we are retrying delivery." }, persists the token, schedules retry attempts according to retry policy, increments delivery-failure metrics, and surfaces operational logs/alerts.

## Edge Cases
   * What happens when a valid verification token already exists and the user requests a resend within 60 seconds?
     - Handling: Do not generate or send a duplicate email; return HTTP 200 with the standard message and include "next_allowed_request_seconds" (X) if the request occurs within a configured debounce window. This conserves email sends and avoids token proliferation.

   * How does the system handle concurrent rapid requests intended to bypass rate limits?
     - Handling: Use atomic counters (Redis INCR with TTL or equivalent) per email/account to track request counts. Ensure the operation to check-and-increment is atomic so only permitted number of requests succeed. Log all concurrent attempts and return HTTP 429 for requests that exceed the limit.

   * What happens if the resend request contains malformed or empty email input?
     - Handling: Validate input server-side (RFC 5322 compliant). For malformed input, return HTTP 400 with explicit error { "error":"invalid_email", "message":"Please provide a valid email address." }. Do not apply generic enumeration behavior for clearly invalid input.

Notes:
- Implementers must ensure the API surface and UI use identical, generic success messaging for unauthenticated calls to avoid account enumeration.
- Rate limit configuration values (per-minute/per-hour/per-day and debounce seconds) must be configurable via feature flags or config and default to: 1 request per 60s debounce, 3 requests per 60 minutes, 10 requests per 24 hours.
- All actions must produce audit logs with sufficient metadata for security investigations (email hashed, requester IP, user-agent, timestamps).

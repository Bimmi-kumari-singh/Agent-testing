## Story ID
   * ID Format: US_<unique seq num>
   * Story ID: us_006

## Story Title
   * Authentication Service Scaffold

## Description
  * As a developer, I want a minimal authentication microservice scaffold (REST surface, healthcheck, structured logging, and basic observability endpoints) so that feature teams can begin implementing authentication features quickly and consistently across environments.

## Acceptance Criteria
  1. Given the service repository is checked out and dependencies installed, When the service is started locally (docker-compose or local runner), Then the service process must start without fatal errors and expose the API port configured in environment (default 8080).
  2. Given the service is running, When a GET request is sent to /health (or /api/v1/auth/health if base path configured), Then the service returns HTTP 200 with a JSON payload containing { status: 'ok', uptimeSeconds: <number>, dependencies: { db: 'unknown'|'ok'|'degraded' } } and content-type application/json.
  3. Given the service is running, When a GET request is made to the base API path /api/v1/auth, Then the service returns HTTP 200 with a JSON body describing available placeholder endpoints and a version field matching semantic version in code (e.g., { version: '0.1.0', endpoints: [...] }).
  4. Given the service receives any incoming HTTP request, When the request is processed, Then the service emits structured JSON logs to stdout that include timestamp, level, service, request_id (generated if absent), method, path, status_code, and duration_ms, and respects log level set by environment variable LOG_LEVEL (default INFO).
  5. Given the service is running in any environment (dev/stage/prod), When the /metrics endpoint is scraped, Then the service exposes a Prometheus-compatible /metrics endpoint with at least basic metrics (process_up 1/0, request_count_total, request_latency_seconds histogram or summary) or an explicit placeholder metric and the health endpoint is suitable for Kubernetes liveness/readiness probes (status 'ok' -> ready).

## Edge Cases
   * What happens when required environment variables are missing?
     - Handling: Service MUST fail fast with a clear error in logs listing missing variables and exit with non-zero code. README and startup logs must document required environment variables and their default values. A developer mode flag (DEV=true) may allow startup with warnings for local development; this must be explicit and opt-in.
   * How does system handle transient dependency (e.g., database) unavailability?
     - Handling: Service starts in "degraded" mode if it cannot reach optional dependencies. Health endpoint indicates dependency state (e.g., db: 'degraded'), and the service performs background retry with exponential backoff. It must not crash-loop; logs must show retry attempts and eventual success/failure.
   * What happens when incoming requests are malformed or route does not exist?
     - Handling: For malformed JSON, return HTTP 400 with JSON error body { error: 'Malformed request', code: 'INVALID_PAYLOAD' }. For unknown routes return HTTP 404 with JSON { error: 'Not Found', code: 'NOT_FOUND' }. All error responses must be logged with request_id and sanitized error details (no sensitive data).

Notes / Implementation Guidance (non-normative):
- Provide repository skeleton with standard directories (cmd/, internal/, pkg/, configs/, scripts/, deployments/) and a clear README with run, build, test, and deploy instructions.
- Include Dockerfile and a minimal docker-compose.override.yml for local dev that starts the service and a lightweight mock DB (sqlite or a stub service).
- Include health (/health), readiness (/ready) or combined endpoint and /metrics; base API prefix /api/v1/auth.
- Implement request-id middleware that checks X-Request-ID header, generates one if missing, and returns it in X-Request-ID response header.
- Use structured logging (JSON) with configurable LOG_LEVEL; do not log secrets or full request bodies.
- Provide CI smoke job that builds the container image and runs a basic healthcheck against a running container (documented but CI job can be in a follow-up story if CI infra not yet present).
- Keep scope limited to the scaffold; do not implement authentication logic (login, tokens) in this story.

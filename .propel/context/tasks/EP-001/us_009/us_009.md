## Story ID

   * ID: us_009

## Story Title
   * Expired/Invalid Token Handling and User Guidance

## Description
  * As an end user who clicks an email verification link, I want the system to detect expired or invalid verification tokens and present clear, actionable messages and options (resend verification and support), so that I can complete account verification or get help quickly without confusion.

## Acceptance Criteria
  1. Given a verification link with an expired token, When the verification endpoint is called, Then the system returns HTTP 410 Gone (API) and the UI displays a message: "This verification link has expired. Request a new verification email to continue." The UI shows a prominent "Resend Verification Email" button, logs the event with user_id and token_id, and does not activate the account.
  2. Given a verification link with an invalid token (malformed, tampered, or not found), When the verification endpoint is called, Then the system returns HTTP 400 Bad Request (API) and the UI displays: "This verification link is invalid. Request a new verification email or contact support." The UI presents "Resend Verification Email" and "Contact Support" CTAs, and the invalid attempt is recorded in security logs with reason code.
  3. Given a verification link for an already-activated account, When the verification endpoint is called, Then the system returns HTTP 200 OK and the UI displays: "Your email is already verified." The UI provides CTAs: "Sign In" and "Go to Home", and no resend option is shown.
  4. Given a user clicks "Resend Verification Email" from the expired/invalid flow, When the user confirms the resend, Then the system calls the resend API which returns HTTP 202 Accepted, the UI displays: "A new verification email has been sent to <email>." The resend action must be subject to rate limits (e.g., max 5 resends per 24 hours) and the UI disables the resend button for the configured cooldown period with a visible countdown timer.
  5. Given the email provider fails during a resend attempt, When the resend API experiences an external service error, Then the system returns HTTP 503 Service Unavailable (or appropriate error), the UI displays: "We couldn't send the verification email right now. Please try again later or contact support." The system must log the provider error details and enqueue a retry according to retry policy (with exponential backoff), and surface a follow-up status in admin logs/alerts.

## Edge Cases
   * What happens when a token expires exactly at verification-time?
     - Handling: The token check must use the canonical expiry timestamp at verification processing time. If expiry_timestamp <= now, treat as expired (410) and present expired flow. Ensure atomic DB check and transition so the account is not activated by borderline race conditions.
   * How does the system handle repeated resend attempts or abuse from the same account or IP?
     - Handling: Apply rate-limiting rules (e.g., per-account: 5 resends / 24 hours; per-IP threshold with CAPTCHA escalations). When limits are exceeded, return HTTP 429 Too Many Requests and show a message with cooldown time and a link to contact support. Suspicious patterns should be logged and flagged to security monitoring.
   * What happens when the account is deleted, suspended, or locked when token verification is attempted?
     - Handling: If account is deleted -> respond with user-friendly "Account not found" and provide link to register. If account is suspended/locked -> display "Account is locked. Contact support" and do not offer resend. All cases should include relevant admin logging and not reveal unnecessary internal state.

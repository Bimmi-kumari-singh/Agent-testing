## Story Title
   * Initial Database Schema & Migrations

## Description
  * As a Platform Engineer, I want an initial, versioned database schema and migration tooling for the authentication service, so that user and authentication data can be stored securely and schema changes can be applied, tested, and rolled back safely across development, staging, and production environments.

## Acceptance Criteria
  * Given a fresh database instance (dev/staging), When the team runs the prescribed migration command (e.g., migrate apply/local-run), Then the following tables exist with the described constraints and indexes:
    - users: id (PK UUID), email (unique, indexed), display_name, created_at (timestamp with timezone), updated_at
    - auth_credentials: id (PK UUID), user_id (FK -> users.id), password_hash, password_version, created_at
    - email_verifications: id (PK UUID), user_id (FK), token_hash, expires_at, used_flag
    - password_resets: id (PK UUID), user_id (FK), token_hash, expires_at, used_flag
    - sessions: id (PK UUID), user_id (FK), refresh_token_hash, expires_at, created_at, last_used_at
    - roles: id (PK), name (unique)
    - user_roles: id (PK), user_id (FK), role_id (FK), assigned_at
    - audit_logs: id (PK), actor_id (nullable FK), action, object_type, object_id, created_at, metadata (jsonb)
    And primary keys, foreign keys, NOT NULLs, unique constraints and appropriate B-tree indexes are present.
  * Given the migration scripts are added to the repository, When CI/CD executes the migration stage during a deployment, Then the migration tool (e.g., Flyway, Alembic, or equivalent) runs migrations in version order, records applied versions and checksums in a migrations table, and fails the pipeline on any error with helpful logs and non-zero exit code.
  * Given an applied migration, When an operator executes the defined rollback/down migration for the latest version, Then schema and constraint state revert to the prior version and automated smoke tests (integration tests in CI) confirm expected table existence and that seeded test rows remain consistent or are restored as defined by the rollback plan.
  * Given two or more processes attempt to apply migrations concurrently (e.g., during blue/green deploy), When migrations are triggered simultaneously, Then the migration system uses an advisory lock or built-in locking mechanism to ensure only one process applies new migrations and other processes detect the lock and wait or exit cleanly without corrupting schema state.
  * Given sample seed/test fixtures run post-migration, When inserting a test user and related auth records, Then:
    - email uniqueness is enforced,
    - password hashes are saved to auth_credentials (no plaintext),
    - inserting a user without required fields fails (NOT NULL),
    - an audit_log entry is created for schema migration runs (migration id, timestamp, initiator).
  * Given the repository contains migrations, When reviewing PRs that add or modify migrations, Then each migration file includes: descriptive name, version id, author, purpose, expected runtime estimate (if >1 minute), and automated lint/validation in CI to verify checksum and simple dry-run compatibility.

## Edge Cases
   * What happens when a migration partially fails due to a runtime error?
     - Handling: Migration tooling must mark the migration as failed in the migrations table (or not mark it as applied), CI must fail, developer runbook must provide steps: restore DB from snapshot (if provided), run provided down-migration if safe, or run a compensating migration. The story must include a short runbook doc snippet with those steps.
   * How does system handle schema changes that require long-running backfills (large-table column addition or type change)?
     - Handling: Require multi-step migration pattern: (1) Add nullable column or new column with default NULL, (2) Deploy application changes that write both old and new columns or tolerate NULL, (3) Run background backfill job outside of migration process (documented and monitored), (4) Add NOT NULL constraint and cleanup in a follow-up migration. CI must reject single-step migrations that change column NOT NULL without prior backfill plan when table row count > threshold (configurable).
   * What happens when environment variables or DB credentials are missing in CI or local dev?
     - Handling: Migration script must validate required env vars on startup and exit with detailed error if missing (list missing vars). CI job must fail fast and include a link to credential/setup documentation. Local helper script should provide a safe dry-run mode that checks connectivity without applying changes.

## Story ID
   * ID Format: US_<unique seq num>
   * ID Provided: us_010

## Story Title
   * Registration Logging, Metrics, and Audit Trail

## Description
  * As an Operations/Compliance engineer, I want registration and email-verification events (creation, email send success/failure, activation, verification failures, resends) recorded as immutable audit events, structured logs, and observability metrics, so that we can monitor health, trigger alerts, investigate incidents, and meet regulatory compliance for recordkeeping and forensics.

## Acceptance Criteria
1. Given a successful user registration request (POST /api/v1/registrations), When the registration is created, Then:
   - a structured audit event with event_type: "registration_created" is persisted to the audit store within 5 seconds of the main transaction commit and contains: user_id (internal id), correlation_id/request_id, timestamp (ISO-8601 UTC), source_ip, client_id, outcome: "created", non-sensitive request metadata; AND
   - a Prometheus-compatible counter "registration_created_total" is incremented; AND
   - a structured JSON log is emitted to centralized logging with same correlation_id and minimal latency.

2. Given the system attempts to send a verification email after registration, When the email provider acknowledges delivery success, Then:
   - persist an audit event event_type: "verification_email_sent" containing user_id, email_address_hash (SHA-256 with application salt), provider_message_id (if available), timestamp, correlation_id; AND
   - increment metric "verification_email_sent_total"; AND
   - observe provider latency in histogram "verification_email_send_latency_ms".

3. Given the email send fails or the provider returns an error, When the failure occurs, Then:
   - persist an audit event event_type: "verification_email_failed" with failure_code (normalized), failure_message (sanitized/truncated), provider_name, timestamp, correlation_id; AND
   - increment "verification_email_failed_total"; AND
   - enqueue a retry per retry policy; AND
   - if the failure rate for a provider exceeds configured threshold (e.g., >5% over 5 minutes), emit an alert (PagerDuty/Slack integration).

4. Given a verification link is consumed and activation succeeds, When the activation flow completes, Then:
   - persist an audit event event_type: "account_activated" containing user_id, activated_by, timestamp, verification_token_id (referenced or hashed), correlation_id; AND
   - increment "account_activated_total" metric; AND
   - the audit event is immutable (no updates) and accessible via read-only Admin Audit API (RBAC enforced) for at least the configured retention period.

5. Given an invalid/expired verification token is used, When the activation attempt fails, Then:
   - persist an audit event event_type: "verification_failed" with normalized error_code ("token_expired"/"token_invalid"), timestamp, correlation_id, user_id if resolvable; AND
   - increment "verification_failed_total" metric; AND
   - return a 400 (invalid) or 410 (expired) response per API contract with no PII in error messages.

Additional measurable checks:
- All audit events must include correlation_id and be written in structured JSON, contain a schema version, and be queryable by user_id and correlation_id.
- End-to-end latency from registration API commit to audit event persisted must be <= 5 seconds under normal conditions (99th percentile); fallback behavior must be exercised in tests.
- Audit store writes must be append-only: attempts to mutate past audit entries should fail with 403 and be logged as an attempted audit integrity violation.

## Edge Cases
- What happens when the audit store or metrics backend is unavailable?
  - The primary user-facing transaction (e.g., create user) must not be blocked permanently. The system will:
    1. Write the primary transaction (user record) atomically.
    2. Enqueue the audit event to a durable retry queue (DB-backed work table or message queue with persistence).
    3. Retry delivery asynchronously with exponential backoff and bounded retry attempts.
    4. If retries exhaust, mark audit event state as "undelivered" and emit a high-severity operational alert; retain the queued item for manual remediation.
    5. All fallback attempts must preserve correlation_id and original event payload (with redactions).
- How does the system handle sensitive data and schema noise?
  - PII (email, password, tokens) is not stored in plaintext in audit events. Emails are stored only as hashed identifiers (salted SHA-256) or masked to support lookup without revealing full PII. Passwords are never recorded. Free-text error messages are sanitized to remove secrets and truncated to a max configured size (e.g., 1024 chars). If a field exceeds allowed size, the stored event contains a truncation indicator and original content is archived only to a protected secure store subject to access controls.
- How are duplicate/retry events deduplicated?
  - Each audit event must include deduplication_key composed of correlation_id + event_type. The audit store enforces uniqueness on that key. Duplicate submissions are idempotently acknowledged; the business flow returns success for idempotent duplicate delivery attempts and does not create additional audit records.

Notes/Implementation Guidance (non-normative):
- Audit event schema: include schema_version, event_type, timestamp_utc, correlation_id, user_id, actor, outcome, metadata (structured map), deduplication_key, redaction_flags.
- Metrics: publish counters (total), histograms for latency, error-rate gauges; expose via Prometheus and attach alerts for error rate and audit backlog.
- Security: ensure audit store access is RBAC controlled and write-only for application components; retention policy configurable (e.g., 7 years) and governed by DR tags; implement WORM or append-only DB capabilities if required by compliance.
- Testing: include unit tests for event composition, integration tests for delivery to mock audit store and metrics exporter, and chaos tests simulating audit store outages to validate retry/fallback.

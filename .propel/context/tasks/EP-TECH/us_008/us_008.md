## Story Title
   * Deployment & Load-Balancing Patterns

## Description
  * As a DevOps/Platform Engineer, I want deployment manifests (Helm examples) and load-balancer patterns for stateless services, so that teams can deploy horizontally-scalable, resilient services with standardized defaults, zero-downtime updates, and clear operational runbooks.

## Acceptance Criteria
  1. Given a staging Kubernetes cluster with cluster credentials and required cluster roles, When a developer uses the provided Helm chart and manifests to deploy the sample stateless service (image: auth-sample:v1) to the staging namespace, Then the deployment includes liveness and readiness probes, resource requests/limits, a Service and Ingress (or cloud LoadBalancer) configured per pattern, and the service responds 200 OK to a health endpoint within 60 seconds of deployment.

  2. Given the provided Helm chart is installed with default HPA settings (target CPU utilization = 60%, minReplicas = 2, maxReplicas = 10), When a sustained CPU load above 60% is applied for 2 minutes during a load test, Then the HPA increases replicas until CPU per pod is near target or maxReplicas is reached, and scaling completes (new pods Ready) within 5 minutes, with metrics emitted to the configured monitoring backend.

  3. Given a Helm upgrade that changes the container image tag, When the chart performs a rolling update using the provided deployment strategy (maxUnavailable and maxSurge configured), Then rolling update completes with zero user-visible downtime (readiness probes pass for new pods before old pods are removed), Helm records a successful revision, and post-deploy smoke tests show ≥99% request success.

  4. Given the Helm chart preflight checks detect cloud provider LoadBalancer provisioning is unavailable or insufficient permissions, When the install/upgrade runs, Then the chart fails gracefully with a clear error and automatically documents/outputs a supported fallback pattern (Ingress backed by NodePort, or external reverse-proxy), including sample manifests and step-by-step remediation guidance in the runbook.

  5. Given the repository includes the manifests, When a developer follows the README and runbook in a fresh environment (local KinD or staging cluster), Then they can fully deploy and validate the sample stateless service (including HPA and ingress) using the provided commands in ≤30 minutes, and rollback instructions are present and validated in a smoke test.

## Edge Cases
   * What happens when HPA reaches maxReplicas (boundary condition)?
     - Handling: The system must emit a metric/alert ("HPA.max_replicas_reached"), documented mitigation steps in the runbook (increase maxReplicas, provision more nodes, add horizontal sharding or queue), and recommend client-side backpressure or API throttling. The Helm chart README must include example thresholds and guidance.

   * How does the system handle cloud LoadBalancer provisioning failure or quota exhaustion (error scenario)?
     - Handling: Helm preflight checks and install hooks detect provisioning failure and abort or mark the release with a clear, human-readable error. The chart includes an alternative manifest set (Ingress + NodePort or external proxy) and automated instructions for switching patterns. The runbook lists provider-specific remediation and permission checks.

   * What happens when pods fail readiness probes repeatedly during autoscaling (error scenario)?
     - Handling: Autoscaler will not expand if pods never reach Ready; alerts are triggered (CrashLoopBackOff / readiness failure). The runbook prescribes steps: check logs/events, disable HPA temporarily, perform Helm rollback if the new release is unhealthy, and run diagnostic commands. CI/CD pipeline should support automated rollback and notification.
